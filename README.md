# HydraDB Query Router (PoC)

Predicts `mode`, `graph_context`, `alpha`, and `max_results` for a HydraDB
`/query` call **before** the call happens, so simple lookups stay fast and
only genuinely complex queries pay for `mode=thinking` (3-5s multi-hop +
rerank). Same admission-control pattern as the SQL query-risk project,
pointed at HydraDB's actual documented pain point: developers currently
have to hand-pick these parameters per query themselves.

## How it decides (in order)

1. **Feature extraction** (`features/extract.py`) cheap, no network/LLM
   calls. Token count, relational/temporal keyword hits, estimated entity
   count, literal-token detection (ticket IDs, error codes), pronoun
   density, session context length.
2. **Hard rules** (`rules/hard_rules.py`) catches obvious cases (an exact
   error code needs `mode=fast, query_by=text`, not semantic search) so the
   model only has to earn its keep on ambiguous queries.
3. **XGBoost classifier** (`model/train.py`) only runs if no rule fired.
   Predicts whether the query needs `thinking` mode.
4. **SHAP explanation** for every model-driven decision, the top 3
   features that drove it, so the decision isn't a black box in the demo.

## Project layout

```
hydra_client/   base.py = interface both clients implement
                mock.py = simulates HydraDB's documented latency behavior
                real.py = thin wrapper around the real POST /query endpoint
features/      extract.py  pre-execution feature extraction
rules/         hard_rules.py obvious-case triage
data/          seed_queries.py real HydraDB cookbook queries + synthetic ones
               bootstrapped_labels.csv generated by eval/bootstrap.py
model/         train.py trains the XGBoost classifier
router.py      ties features -> rules -> model -> decision together
api/           main.py FastAPI app (/route and /route/execute)
eval/          bootstrap.py generates ground-truth labels from real runs
               benchmark.py router vs. naive baselines
```

## Setup (PowerShell, VS Code terminal)

```powershell
cd hydra-router
python -m venv venv
.\venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

If PowerShell blocks the activation script with an execution-policy error,
run this once (only affects your user, not the whole machine):

```powershell
Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
```

## Running it (mock client no API key needed)

```powershell
# 1. Generate ground-truth labels by running seed queries through the mock
python -m eval.bootstrap --client mock

# 2. Train the classifier on those labels
python -m model.train

# 3. See router vs. naive-baseline latency numbers
python -m eval.benchmark --client mock

# 4. Start the API
uvicorn api.main:app --reload --port 8000
```

Then, in a second terminal (or Postman/curl):

```powershell
curl.exe -X POST http://127.0.0.1:8000/route `
  -H "Content-Type: application/json" `
  -d '{\"query\": \"Why did we choose Postgres?\"}'
```

## Switching to the real API once you have a key

```powershell
$env:HYDRA_DB_API_KEY = "your-key-here"
$env:HYDRA_DB_DATABASE = "router_poc_db"   # create this database first via their API

python -m eval.bootstrap --client real --db-name router_poc_db
python -m model.train
python -m eval.benchmark --client real --db-name router_poc_db
uvicorn api.main:app --reload --port 8000   # now uses RealHydraClient automatically
```

No code changes needed `api/main.py` and the eval scripts check for
`HYDRA_DB_API_KEY` in the environment and switch clients automatically.

**Important honesty note:** every number produced with `--client mock` is a
simulated stand-in, calibrated to HydraDB's documented latency figures
(fast <=500ms, thinking 3-5s) it exists so the pipeline can be built and
tested end to end before signing up. Do not put mock-derived benchmark
numbers in front of the founder. Re-run `bootstrap` and `benchmark` with
`--client real` before the demo and use those numbers only.

## Is this a microservice? Python or Go?

**For the PoC: no, keep it a single Python service.** Here's the actual
reasoning, not just a default:

- The whole value proposition is "decide in microseconds so it's cheaper
  than the call you're avoiding." A network hop to a separate service would
  eat into that budget for no benefit at this scale the decision itself
  is a handful of regex checks and one small tree-model inference, which is
  not the kind of workload Go's concurrency model helps with. Go earns its
  place when you're pushing high concurrent I/O or need a small static
  binary for deployment; here the bottleneck (if any) is the *downstream*
  HydraDB call, not the decision.
- XGBoost, SHAP, and the training loop are Python-ecosystem tools. Doing
  the ML in Python and the serving in Go means either re-implementing
  tree-model inference in Go (real work, real bug surface) or running two
  languages for a project that's supposed to demo scope discipline, not
  polyglot infrastructure for its own sake.
- One FastAPI service keeps the whole demo buildable and explainable in a
  weekend, which was the actual constraint you were optimizing for.

**If this got adopted for real** (see scaling section below), splitting the
hot-path decision function into a tiny separate service *could* make sense
but even then I'd reach for a fast Python path (the model is small enough
that raw XGBoost inference is sub-millisecond) before reaching for Go,
unless HydraDB's own stack is already Go and they'd want it native for
maintainability reasons. That's a call to make with them, not before.

## Scaling plan (if HydraDB actually wanted this for their user base)

Right now this is a PoC that decides for one query at a time, called
directly by a developer. Scaling it to sit in front of HydraDB's real
traffic is a different problem with a different shape:

1. **Move from "library you call" to "logic HydraDB runs at the edge of
   `/query`."** The end state isn't a separate service developers have to
   integrate it's a `mode: "auto"` option HydraDB's own `/query` endpoint
   accepts, with this decision logic running inside their existing request
   path. That removes the extra network hop entirely and is the version
   that's actually shippable as a product feature, not a bolt-on.

2. **Replace the bootstrapped mock/small-sample labels with real production
   query logs.** The biggest weakness of this PoC, stated plainly: it's
   trained on ~40 seed queries. At HydraDB's actual scale, they have real
   query logs with real mode selections, real latencies, and (if they track
   it) real downstream signal for whether the answer was good enough. A
   production version retrains on that, not on cookbook examples.

3. **Per-tenant / per-collection model calibration.** A finance-domain
   customer's "simple" query and a coding-agent customer's "simple" query
   don't look the same lexically. At scale this probably becomes either a
   shared base model with a lightweight per-tenant calibration layer, or
   a shared feature set with per-tenant decision thresholds full custom
   models per tenant would be over-engineering for what's fundamentally a
   binary routing decision.

4. **Feedback loop, not a static model.** Log every routing decision plus
   what would have happened on the other path (shadow the untaken mode at
   a low sample rate, not every request) so the model can be retrained
   against outcomes instead of going stale. This is the same idea as
   canary/shadow deployment, applied to a classifier instead of a service
   version.

5. **Latency budget for the decision itself becomes a real SLA, not just a
   nice property.** At real scale, the decision layer sits on the critical
   path of every single query, so it needs its own monitoring (p50/p99 of
   the decision step itself, separate from the downstream HydraDB call) and
   a hard fallback (a rule-only mode with no model call) if inference ever
   regresses never let the thing meant to reduce latency become a new
   source of it.

None of this needs to be built for the demo. It's the answer to the
natural follow-up question a founder would ask if they liked the PoC:
"okay, how would this actually become a feature."
